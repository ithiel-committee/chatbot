# AITuber / AI Chatbot 実装技術サーベイ報告書

AITuber（AIを活用したバーチャルキャラクター・配信システム）およびAIチャットボットの実装における主要なアプローチ、オープンソースリポジトリ、およびイティエル（IThieL）プロジェクトへの具体的な応用提案をまとめました。

---

## 1. 主要なオープンソースプロジェクトとアーキテクチャ

現在、Webベースの3Dキャラクターインタラクションにおいてベンチマークとなる代表的なプロジェクトは以下の通りです。

### ① ChatVRM / LocalChatVRM (pixiv)
*   **概要**: ブラウザ上でVRMキャラクターと対話できるオープンソースのデモアプリ。
*   **特徴**:
    *   **フロントエンドファースト**: `React` + `three-vrm` で構築され、ブラウザ単体でVRMの表示・表情アニメーションが完結。
    *   **マルチTTS対応**: OpenAI TTSやElevenLabs、KOEIRO APIなど複数の音声合成サービスと連携。
    *   **LocalChatVRM**: Google I/O 2025にて公開されたWebGPU / WebNNを用いた**完全ブラウザローカル動作**のモデル。
*   **参考ポイント**: VRMの表情遷移（Emotion Manager）と音声再生・リップシンクの連携コードが極めて洗練されている。

### ② AITuberKit (tegnike)
*   **概要**: ChatVRMをベースに、YouTube配信やマルチLLM・音声合成エンジン対応を施したオールインワンツールキット。
*   **特徴**:
    *   **多様なTTS対応**: `VOICEVOX`、`Style-Bert-VITS2`、`ElevenLabs`、`AquesTalk` を容易に切り替え可能。
    *   **会話継続モード**: ユーザーからの入力がない間も、AIが自発的に思考・発話する仕組み。
    *   **配信対応**: OBS WebSocketを介した字幕自動挿入や、YouTube Liveコメントのリアルタイム監視。
*   **参考ポイント**: ストリーミングテキストを効率的にさばくためのキュー管理と、字幕表示の同期設計。

## 2. リアルタイム発話（ストリーミング）と音声合成の最適化

LLM（Gemini/ChatGPTなど）のストリーミング応答（逐次トークン出力）に合わせて自然に発話させるには、**テキストの動的分割**が必須のテクニックとなります。

### なぜ分割が必要か？
1.  **低遅延（レイテンシ削減）**: 全文が生成されるのを待ってから音声合成APIを叩くと、数秒〜数十秒のタイムラグ（沈黙）が発生し、対話のテンポが損なわれます。
2.  **音声合成の品質**: 一定以上の長文を一度に音声合成にかけると、モデルのコンテキスト制限から不自然なイントネーションになったり、句読点での「間」が崩れたりします。

### 一般的なストリーミング分割ロジック
多くのAITuber実装では、LLMから流れてくる文字ストリームを監視し、以下の条件で音声合成のジョブに回します（キューイング）。

```
[LLM文字ストリーム] ──> [バッファ監視] 
                           │
                           ├──> 句読点（「。」「！」「？」）を検出
                           ├──> または 改行コード を検出
                           └──> かつ 最小文字数（例：15文字以上）を満たしている
                                   │
                                   ▼（テキストをカット）
                       [音声合成エンジン (TTS)] ──> [音声再生キューに追加]
```

*   **安全弁としての強制分割**: 句読点がない長文が生成された場合に備え、最大文字数（例：80文字）に達した時点で強制的に分割を入れます。

## 3. VRMモデルのリップシンク（口パク）実装手法

ブラウザ上で音声とVRMモデルの口の動き（BlendShape / Expression）を同期させるアプローチには、主に2つの手法があります。

### 手法A: 音量（RMS）連動方式（軽量・汎用的）
Web Audio APIの `AnalyserNode` を使用し、再生中の音声データのリアルタイム音量を抽出して、VRM of `Aa（「あ」の口）` のウェイトにマッピングします。

*   **メリット**: 実装が非常に簡単で、VOICEVOXやGoogle TTSなどあらゆる音声合成エンジンに対して追加データなしで動作する。
*   **デメリット**: 母音（あ・い・う・え・お）の区別がなく、音量に比例してただ口が縦に開閉するだけになる。

### 手法B: 音素（タイムスタンプ）連動方式（高精度）
VOICEVOXやStyle-Bert-VITS2などのエンジンから、音声データと一緒に「どのタイミングでどの音素（母音）を発音するか」という時間情報（JSON）を取得し、再生時間（`AudioContext.currentTime`）と同期させて口の形（`Aa`, `Ih`, `Uu`, `Ee`, `Oh`）を切り替えます。

*   **メリット**: 「こんにちは」であれば「o - i - i - a」のように正確な口の形になり、極めてリアルになる。
*   **デメリット**: 音声合成側が音素情報（タイムスタンプ）を返せる必要がある。また、Web Audio APIの正確な時間管理ロジックのコーディングが必要。

## 4. ユーザー音声の音声認識 (Speech to Text: S2T)

AIチャットボットと対話する手段として、テキストだけでなくユーザー自身の声からリアルタイムに文字起こしして入力する処理もAITuberやインタラクティブUIでは主流です。

### ① Web Speech API (ブラウザ標準)
*   **特徴**:
    *   外部のAPIキーが不要、無料でブラウザだけで動作。
    *   ChromeやEdgeなどのChromium系ブラウザで動作が極めて安定。
    *   `recognition.continuous = true` に設定することで、聞き取りを維持したまま会話ができます。
*   **制御のコツ**:
    *   **無音区間検出 (Silence Detection)**: ユーザーが喋り終わったタイミング（`onresult` で一定秒数テキストが追加されない状態）を検知して自動でLLMへ送信リクエストを投げるトリガーを設けると、ボタン不要のハンズフリー対話が実現します。

### ② クラウドAPI (Whisper, Groq等)
*   **特徴**:
    *   マイクからの入力を `MediaRecorder` で録音し、数秒ごとに細切れのバイナリをWhisper API等に投げて文字起こしします。
    *   ブラウザ依存がなく、あらゆる環境で極めて高い文字起こし精度を提供。
    *   GroqのWhisper API等を使えば数十ミリ秒単位の超高速な文字起こしが可能です。

## 5. キャラクターの表情、ポーズ、アイドリングの動的制御

3Dモデルに生命感を吹き込むため、AIキャラクターのポーズや表情は単なる固定ポーズではなく、時間経過による揺らぎ（アイドリング）や、発話の感情に応じた動的遷移が求められます。

### ① 自然な動き（アイドリング・まばたき・視線）
*   **アイドリング (Idle Animation)**: `Math.sin(clock.getElapsedTime())` などの周期関数を使い、呼吸に合わせて肩や呼吸ボーンをわずかに（例：最大±2度程度）上下させたり、首を少し左右に揺らすことで、「静止している不気味の谷」を回避します。
*   **自動まばたき (Blink)**: 定期タイマー（数秒〜十数秒の間でランダム）を仕掛け、`Blink` のウェイトを `0.0 -> 1.0 -> 0.0` へ素早く（0.2秒程度）補間させることで、本物の人間らしいまばたきを作ります。
*   **視線追従 (LookAt)**: `@pixiv/three-vrm` の `vrm.lookAt` にカメラ（ユーザー視点）やマウス位置をターゲットとしてバインドします。キャラクターがこちらを追視するようになり、愛着や親近感が劇的に高まります。

### ② LLM（AI）と連動した「感情タグ」の抽出
LLMにシステムプロンプトで「文頭に必ず感情タグを付けて出力すること」を指示（Few-Shot）し、出力されたテキストをフロントエンドでパースして表情（ブレンドシェイプ）や待機モーションを変更します。

*   **プロンプト例**:
    > 会話をする際は、文頭に `[happy]` や `[sad]` などの感情タグを必ず1つだけ含めて出力してください。
*   **パース処理例**:
    ```javascript
    // 受信した文字の文頭からタグを抽出する正規表現
    const match = text.match(/^\[(happy|angry|sad|relaxed|neutral)\]/);
    if (match) {
      const emotion = match[1];
      setTargetExpression(emotion); // 表情の切り替え
      const cleanText = text.replace(/^\[.*?\]/, ""); // 表示用のテキストからタグを削除
    }
    ```

## 6. Web上での表示・レイアウト (Canvas と UI の階層制御)

画面全体に3Dキャラクター（Canvas）を敷きつつ、その前面に操作パネルやチャットエリア（HTML）を配置する際のフロントエンド設計です。

### ① CSSによる重ね合わせと pointer-events の排他制御
*   **階層構造**: WebGLの `canvas` を `position: absolute; z-index: 1;` として最背面に置き、チャットコンテナ（`.text-area-container`）を `z-index: 10;` などで手前に重ねます。
*   **クリック透過処理**:
    *   3Dキャラクター自体のドラッグや回転操作、および視線追従を有効にしたまま、手前のHTMLの邪魔な透明空間をクリック透過させたい場合、UIコンテナに `pointer-events: none;` を指定します。
    *   チャットの入力欄（`input`）やボタンなどの「触りたい要素」だけに対して、ピンポイントで `pointer-events: auto;` を設定することで、3Dシーンの操作とHTML UIの操作が干渉せずに共存できます。

```css
/* UIコンテナ（最前面の親レイヤー） */
#ui-overlay {
  position: absolute;
  top: 0; left: 0; width: 100vw; height: 100vh;
  z-index: 10;
  pointer-events: none; /* 下のCanvasにクリックを通す */
}

/* 入力エリアやボタン（触れる子要素） */
.interactive-ui {
  pointer-events: auto; /* この要素だけはクリック可能にする */
}
```

## 7. バックエンドシステム（APIキーの隠蔽と履歴DB）

AITuberやAIチャットボットを一般公開（Web公開）するにあたり、バックエンドの設計はセキュリティおよびパフォーマンスの観点から非常に重要です。

### ① APIキーの隠蔽（セキュア設計）
*   フロントエンド（ブラウザ）から直接 OpenAI や Gemini の API キーを埋め込んでリクエストを投げるのは**絶対厳禁**です（ソースコードを解析されてAPIキーが盗まれるリスクがあります）。
*   Node.js (Express) や Python (FastAPI) などのバックエンドを経由させ、環境変数（`.env`）でAPIキーを保持した上で、プロキシとしてリクエストを受け渡しします。

### ② セッション管理と履歴DB（SQLite）
*   **SQLiteの採用**: 軽量で独立したファイルデータベースである SQLite は、会話履歴やセッションごとのステータス（ユーザーの親密度、感情記憶など）を永続化するのに最適です。
*   対話リクエストが届いた際、DBから「同一セッションの直近3〜5回のやり取り」をロードしてプロンプトのコンテキスト（履歴）に注入することで、脈絡のある対話を維持します。

### ③ 通信方法の最適化 (Server-Sent Events: SSE)
*   HTTPリクエストでLLMに問い合わせると、返答がすべて完成するまでフロントエンドが待たされるためレスポンスが遅く感じられます。
*   `Server-Sent Events (SSE)` または `WebSocket` を使って、バックエンドがLLMから受信したテキストをリアルタイムでフロントエンドに流し込む（ストリーム伝送）設計にすることで、極めてテンポの良い対話感を生み出せます。
